## 회고

- 오늘 내가 맡은 일: 사용자가 나중에 고민할 수 있도록 Jarvis 개선 항목을 별도 문서로 남기고, Markdown 작성 언어 원칙을 프로젝트 지침에 반영했다.
- 실제로 한 일: 개선 백로그를 `docs/`에 만들고, `AGENTS.md`에 사용자가 읽는 Markdown은 가급적 한글로 작성한다는 원칙을 추가했다.
- 어려웠던 지점: 영어 고유명사와 코드 식별자는 한글화하면 오히려 정확성이 떨어질 수 있어 예외 규칙을 함께 적었다.
- 판단을 바꾼 순간: 단순히 작업 폴더 산출물로만 두면 나중에 찾기 어려울 수 있어, 장기 참조 문서인 `docs/`에도 별도 백로그를 두기로 했다.
- 인간 대표 또는 다른 에이전트에게 받은 피드백: 사용자가 한글 Markdown 저장 원칙을 명시했다.

## 학습

- 새로 배운 점: Jarvis 문서는 실행 결과뿐 아니라 사용자의 운영 선호도도 명시적으로 보존해야 다음 작업에서 일관성이 생긴다.
- 다음에 반복하지 말아야 할 실수: 사용자가 읽을 문서에 불필요하게 영어 제목과 영어 본문을 기본값으로 쓰지 않는다.
- 다음에도 재사용할 수 있는 패턴: 장기 참고용 문서는 `docs/`, 작업별 증거와 브리프는 `work-requests/`에 둔다.
- 지혜 승격 후보: 사용자가 읽는 Jarvis Markdown 산출물은 한글을 기본으로 하고, 코드/명령어/고유명사는 원문 표기를 유지한다.
