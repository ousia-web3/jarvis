## 회고

- 오늘 맡은 일은 짧은 사용자 요청을 전체 서비스 설계 문서로 확장하는 것이었다.
- 사용자가 "MVP 아니고 전체 서비스"라고 명확히 말했기 때문에, MVP 범위 축소 대신 전체 청사진과 Wave 실행 계획을 만들었다.
- 유튜브 영상 제목과 공개 메타데이터는 여행 욕구를 보여주는 힌트이지만 실제 국적별 선호를 증명하지는 않는다. 그래서 모든 국적/동행 추천은 `가설`로 분리했다.
- 이미지가 중요한 서비스지만 권리 리스크가 크므로, 공통 배너 placeholder와 이미지 권리 상태 필드를 핵심 설계 요소로 승격했다.
- 추가 요청에서 콘텐츠 관리 DB 효율성이 중요해졌고, 단순 테이블 나열보다 운영 큐와 읽기 모델까지 포함해야 한다고 판단했다.
- 이후 추가 요청에서 Trip Ontology Dashboard가 빠져 있음을 보강했고, 여행 콘텐츠는 목록/검색뿐 아니라 관계 근거를 운영자가 볼 수 있어야 한다고 판단했다.
- 최종 검토 요청에서 온톨로지가 W3처럼 보이는 우선순위 충돌을 발견했고, W0/W1 선행 트랙으로 올리는 것이 서비스 전체 일관성에 더 맞다고 판단했다.
- Supabase 스키마 분리 요청에서 단일 DB 안에 여러 Git 프로젝트가 공존하는 구조라면 `public` 사용을 지양하고 프로젝트 전용 schema를 명확히 고정해야 한다고 판단했다.

## 학습

- 영상 기반 여행정보 서비스는 `영상 영감 -> 구조화 여행정보 -> 코스 후보` 흐름으로 설계하면 저작권 리스크와 사용자 가치의 균형이 좋다.
- 외국인 대상 서비스라도 초기 콘텐츠가 한국어라면 "외국인을 초대하는 국내 거주자"를 중요한 초기 사용자로 잡을 수 있다.
- 국적별 추천은 고정관념을 피하기 위해 추천 이유, 대체 코스, 행동 데이터 보정 체계를 함께 설계해야 한다.
- 여행 콘텐츠 DB는 `정규화 원장 + JSONB staging + 검색 읽기 모델 + CMS 작업 큐` 조합이 관리 효율과 확장성의 균형이 좋다.
- 온톨로지 기반 추천 근거는 `Evidence`와 운영자 승인 상태를 가져야 하며, 승인 전 relation assertion은 공개 서비스에 반영하지 않는 것이 안전하다.
- 온톨로지는 대시보드 기능 하나가 아니라 검색 facet, CMS 공개 조건, 추천 이유, 권리 영향 분석의 공통 언어이므로 선행 설계해야 한다.
- Supabase `GRANT`와 Data API expose는 접근 가능성을 열 뿐이며, 실제 보안 경계는 RLS 정책과 service role 분리로 완성해야 한다.
