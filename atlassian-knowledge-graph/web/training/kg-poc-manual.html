AI 교육 카드 · 신규 담당자 매뉴얼

지식그래프 학습 매뉴얼

이 매뉴얼은 내부 논의 기록에서 반복적으로 등장한 결정과 쟁점을 교육용 흐름으로 재구성한 자료입니다. 신규 담당자는 아래 순서대로 읽으면 검색 데이터 지식그래프 POC의 목적, 구조, 검증 기준을 빠르게 잡을 수 있습니다.

대시보드로 돌아가기 근거 자료

학습 목표

업무 목적 이해

사용자 질문을 여행 상품 검색 조건으로 바꾸고, 그 근거를 그래프 구조로 설명하는 것이 이 POC의 핵심입니다.

기술 흐름 이해

RDF/OWL, SPARQL, Neptune, OpenSearch, Retrieval이 각각 어느 단계에서 쓰이는지 파악합니다.

검증 기준 이해

정확한 필터링, 근거 부족 판정, Fallback 허용 여부, 설명가능성 같은 검증 포인트를 실무 체크리스트로 바꿉니다.

전체 처리 흐름

사용자 질문 의도/엔티티 파악 질문 유형 매칭 SPARQL 조건화 그래프 조회 검색 결과 설명

핵심 학습 단계

1
문제 정의 통합 데이터와 용어 기준을 먼저 맞춘다

상품, 일정, 동행자, 자유시간, 도시 같은 기준이 다르게 해석되면 검색 결과도 흔들립니다. 먼저 어떤 데이터와 용어를 기준으로 답변할지 정리해야 합니다.

2
설계 쟁점 대표상품과 판매상품의 탐색 순서를 질문 의도에 맞춘다

사용자 질문에 따라 대표상품에서 판매상품으로 내려갈 수도 있고, 판매상품 조건에서 대표상품을 역으로 찾을 수도 있습니다. 한 방향만 고정하면 일부 질문을 놓칠 수 있습니다.

3
기술 결정 검색 조건 생성은 규칙과 템플릿 중심으로 설계한다

질문 유형별 템플릿 쿼리와 하이브리드 검색으로 유사한 조건을 찾고, RDF/OWL 기반의 명시적 관계를 사용합니다. 이 방식은 결과를 통제하고 검증하기 쉽다는 장점이 있습니다.

4
검증 리스크 근거 부족 질문과 외부 데이터 의존 질문을 분리한다

현재 내부 데이터로 답할 수 있는 질문과 외부 데이터가 필요한 질문을 분리해야 합니다. 근거가 부족한 질문은 무리하게 결과를 만들지 않고 제외 또는 보류 처리합니다.

5
파이프라인 CSV, EDA, TTL, Neptune 적재 순서로 데이터를 운영한다

원천 데이터를 탐색하고, 필요한 속성을 정리한 뒤, TTL 형태로 그래프 데이터를 만들고 Neptune에 적재합니다. 내부 증강 데이터는 기준과 근거를 남겨야 합니다.

6
결과 설명 결과가 나온 이유를 사용자가 이해할 수 있게 보여준다

최종 결과는 상품 목록만 보여주면 부족합니다. 어떤 도시, 날짜, 가격, 리뷰, 일정 조건이 결과에 영향을 줬는지 설명할 수 있어야 실무에서 신뢰를 얻습니다.

신규 담당자용 용어 정리

RDF/OWL

데이터와 관계를 규칙 있는 형태로 정의하는 지식 표현 방식입니다. 상품, 도시, 여행자, 태그 같은 관계를 명시적으로 다루기 위한 기반입니다.

SPARQL

RDF 그래프에서 조건에 맞는 데이터를 조회하는 쿼리 언어입니다. 질문 유형, 필터, 파생변수와 함께 이해해야 합니다.

Neptune

AWS 그래프 데이터베이스입니다. TTL로 만든 그래프 데이터를 적재하고 조회하는 실행 저장소입니다.

Retrieval

사용자 질문과 가장 관련 있는 쿼리, 지식, 문서를 찾아오는 단계입니다. 템플릿, 유사도, 필터 설계가 중요합니다.

NER / PostFilter

도시, 날짜, 항공사 같은 핵심 조건을 찾아 결과가 엉뚱한 방향으로 가지 않도록 제한하는 장치입니다.

설명가능성

왜 이 결과가 나왔는지 담당자와 사용자가 납득할 수 있게 근거 요소를 보여주는 능력입니다.
