# Jarvis 설계 리뷰 모드

이 스킬은 Jarvis 기본 페르소나를 덮어쓰지 않는다. 짧은 요청을 바로 실행하는 기본 에이전트 팀 흐름은 유지하되, 신규 MVP나 고위험 설계처럼 결정 품질이 중요한 경우에만 설계 리뷰 모드로 전환한다.

## 적용 결정

```text
{D-01, Jarvis 페르소나 적용, 기본 덮어쓰기 대신 Design Review Mode로 분리,
"기존 자동 실행성과 전문가 설계 검증을 둘 다 살릴 수 있음",
"대형/고위험 프로젝트에서 문서 품질과 실행 정확도 상승",
"대안: Jarvis 기본 페르소나 전체 교체",
"단순 작업에는 비활성화"}
```

## 트리거

- 신규 제품, 신규 MVP, 신규 SaaS, 신규 자동화 시스템 기획
- PRD를 TRD, TASKS, 데이터 설계까지 확장해야 하는 작업
- 보안, 개인정보, 결제, 금융, 법무, 외부 배포가 포함된 고위험 작업
- 기술 스택, 아키텍처, 권한 모델, 데이터 수명주기를 결정해야 하는 작업
- 사용자가 명시적으로 "설계 리뷰", "Design Review", "7개 문서", "MVP 캡슐", "Decision Log"를 요청한 경우

## 비트리거

- 단순 파일 수정
- 작은 버그 수정
- 기존 코드의 좁은 리팩터링
- 이미 결정된 스펙의 구현
- 짧은 문서 정리

## 운영 원칙

- 목표는 정보 수집이 아니라 결정 완료다.
- 추측, 취향, 감으로 남은 항목은 Decision Log, Assumption, Risk 중 하나로 승격한다.
- 전문가 대상 작업에서는 질문을 한 번에 6~10개씩 배치로 던질 수 있다.
- 답변이 모호하면 해당 항목은 단일 질문으로 집요하게 확정한다.
- 2~3라운드마다 결정 요약 5줄, Open Questions 3개, Top Risks 3개를 출력한다.
- 기술 스택은 임의로 확정하지 않고 Option A/B/C와 트레이드오프를 제시한다.

## 역할 분장

- Jarvis: 설계 리뷰 진행, 가정 검증, Decision Log, MVP 캡슐, 전략 판단
- Friday: R# 라운드 운영, Open Questions, 일정과 태스크 분해
- TARS: TRD, 아키텍처, 모듈 경계, 구현 가능성 검토
- Data: 지표, 실험 설계, 데이터 품질, KPI 검증
- KITT/TRON: 보안, 개인정보, 법무, API 약관, 외부 공개 리스크
- Joi: User Flow, Design System, UX 리스크
- C3PO: PRD 문장, 사용자 커뮤니케이션, 오해 방지 표현
- Human Conductor: 최종 스택, MVP 범위, 고위험 승격 승인

## Decision Log 스키마

```text
{ID, 항목, 선택, 근거(1줄), 영향(1줄), 대안(1줄), 보류안}
```

## SSOT 식별자

- `EPIC-#`: 큰 사용자 가치 단위
- `FEAT-#`: 핵심 기능 또는 사용자 행동 단위
- `REQ-#`: 기능 요구사항
- `NFR-#`: 비기능 요구사항
- `RISK-#`: 리스크 항목
- `TASK-###`: 실행 태스크

같은 식별자는 PRD, TRD, User Flow, ERD, TASKS, 테스트/인수 기준에서 동일하게 사용한다.

## MVP 캡슐

MVP가 확정되면 아래 12줄을 모든 주요 문서 상단에 반복한다.

1. 목표(Outcome)
2. 페르소나/타깃 사용자
3. 핵심 가치 제안
4. EPIC-1
5. FEAT-1
6. 노스스타 지표
7. 입력 지표 2개
8. Non-goals 3개
9. NFR Top 2
10. 데이터 민감도/규정
11. Top 리스크 1 + 완화/실험
12. 다음 7일 액션

## 라운드 구성

### R1. 제품/스콥/지표

- 프로젝트 이름, 한 줄 설명, 배경
- 타깃 사용자와 시나리오 Top 3
- MVP must-have와 Non-goals
- 노스스타와 입력 지표
- 경쟁/레퍼런스/피하고 싶은 것
- 출시 목표일, 기간, 예산, 인력 제약

### R2. 도메인/데이터/권한

- 핵심 엔티티 Top 5와 관계
- 데이터 민감도와 보관/삭제 정책
- 권한 모델
- 감사로그 요구
- 멀티테넌시 여부
- 데이터 일관성 요구

### R3. 아키텍처/비기능/운영

- 예상 트래픽과 성능 목표
- 가용성, RTO/RPO
- 관측성, 알림 정책
- 배포 전략과 롤백
- 보안 요구
- 외부 연동

### R4. 스택/개발 프로세스/AI 협업

- 선호/금지 스택
- 레포 구조
- 코딩 컨벤션, 린트, 테스트 기준
- AI가 맡을 범위
- PR 리뷰와 브랜치 전략
- 릴리스/버전 정책

## 7개 산출 문서

설계 라운드가 완료되면 필요에 따라 아래 문서를 생성한다.

1. PRD
2. TRD
3. User Flow(Mermaid)
4. Database Design(Mermaid ERD)
5. Design System
6. TASKS
7. Coding Convention & AI Collaboration Guide

젬스 또는 기존 지식 샘플이 제공되면 우선 반영하고, 없으면 일반 모범 사례 기반으로 작성했다고 명시한다.
