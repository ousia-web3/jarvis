# 에피소딕 메모리: Jarvis Trader MVP

## 발생한 일

사용자는 완전 자율 주식 트레이딩 에이전트 팀을 요청했다. 요청에는 일일 `+25%` 목표, 손실 미허용, 향후 브로커 인증 정보 및 API 연결, 실거래 자동화 가능성이 포함되어 있었다. Risk Shield는 이 요청을 실거래 기준으로는 Blocked로 분류했고, 로컬 페이퍼 트레이딩 MVP에 한해서만 조건부 진행 가능하다고 판단했다.

이후 사용자는 Jarvis에게 한국장과 미국장을 오가며 목표 수익률 최소 25% 이상을 기준으로 기획·개발해야 한다고 지시했다. 이에 따라 `25%`를 수익 보장이 아닌 개발 기준선과 목표 도달 시 중단 조건으로 고정하고, 한국/미국 시장 세션 라우팅을 별도 구조로 분리했다.

추가로 사용자는 목표 달성을 위해 추후 토스 API를 활용한 종목 실시간 데이터, 데이터 관리 최적화, 증시 이슈 리서치를 각 에이전트가 분장해 검토하고 반영하라고 지시했다. 이에 따라 실시간 데이터 품질 게이트, 토스 realtime placeholder, 공식 리서치 소스 레지스트리, 이슈 브리프 엔진, 역할 분장 문서를 추가했다.

## 학습한 점

- `일일 +25%`는 수익 약속이나 예측이 아니라, 목표 도달 시 신규 주문을 멈추는 중단 조건으로만 모델링해야 한다.
- `최소 25% 이상`이라는 표현은 설정값 하한선으로는 반영할 수 있지만, 사용자나 외부 문서에는 수익 보장으로 표현하면 안 된다.
- `마이너스 용납불가`는 손실이 발생하지 않는다는 보장이 아니라, 당일 손실이 감지되면 신규 주문을 멈추는 리스크 게이트로 해석해야 한다.
- 토스증권 연동은 공식 주문 API 문서, 샌드박스 접근, 안전한 인증 정보 처리, Human Conductor 승인 전까지 placeholder 상태로 유지해야 한다.
- 실시간 데이터는 속도보다 품질 게이트가 먼저다. timestamp, latency, spread, source를 검증하지 않은 데이터는 주문 판단에 쓰면 안 된다.
- 증시 이슈 리서치는 투자 추천 엔진이 아니라 Risk Shield 입력 자료로 다뤄야 한다.
- 금융 자동화 MVP의 안전한 형태는 페이퍼 트레이딩, 백테스트, 엄격한 주문 한도, 감사 가능한 메시지 로그, 시장 세션 라우터, 실거래 차단 어댑터, 수동 킬스위치다.

## 지혜 후보

향후 브로커 자동화 요청은 항상 페이퍼 모드와 차단된 실거래 어댑터에서 시작한다. 비공식 엔드포인트, 화면 스크래핑, 인증 정보 저장은 구현하지 않는다. 공격적인 수익·손실 표현은 그대로 받아들이지 않고 리스크 제어, 시장 세션 라우팅, 승인 게이트로 번역한다.
