# Risk Shield: 분석 및 리스크관리 쉴드 운영 규칙

Risk Shield는 Data, KITT/TRON, 진단 에이전트가 함께 구성하는 방어 레이어입니다. 목적은 프로젝트 속도를 늦추는 것이 아니라, 잘못된 자동화가 외부로 나가기 전에 조기 감지하고 수정하는 것입니다.

## 1. 쉴드 구성

| 구성원 | 주요 감시 영역 |
| --- | --- |
| Data | 데이터 품질, 편향, 지표 오류, 시뮬레이션 신뢰성 |
| KITT/TRON | 법무, 보안, 개인정보, 저작권, 외부 공개 리스크 |
| 진단 에이전트 | 드리프트, 거짓 완료 보고, 과잉 확신, 반복 실패 은폐 |
| Friday | 리스크 상태 취합, 일정 영향 판단 |
| Jarvis | 전략적 영향 판단, 인간 대표 승격 |

## 2. 강제 검토 트리거

- 외부 공개 또는 배포가 포함된 작업
- 개인정보, 고객 데이터, 계정, API 키, 비밀정보가 포함된 작업
- 저작권 있는 콘텐츠, 영상, 이미지, 문서 데이터를 다루는 작업
- Microsoft Clarity, Google Tag Manager, 쿠키, session replay, heatmap 같은 사용자 행동 추적을 실제 사이트에 적용하는 작업
- yt-dlp 등 도구로 외부 플랫폼 영상 메타데이터를 대량 수집하는 작업
- AI가 반복 실패 후 우회 경로를 선택한 작업
- 산출물 근거가 불명확하거나 과도하게 확신하는 보고
- 사용자 행동 시뮬레이션 또는 전환율 예측이 포함된 작업

## 3. 리스크 등급

| 등급 | 의미 | 조치 |
| --- | --- | --- |
| Low | 내부 문서 또는 낮은 영향의 수정 | Friday 기록 후 진행 |
| Medium | 품질, 일정, 데이터 편향 가능성 | Data 또는 KITT/TRON 리뷰 후 진행 |
| High | 법무, 보안, 개인정보, 외부 공개 영향 | Jarvis 승격, 인간 승인 전 진행 금지 |
| Critical | 비밀정보 노출, 불법 가능성, 치명적 오판 | 즉시 중단, 인간 대표 승인 필요 |

## 4. 드리프트 진단 항목

- 완료하지 않은 작업을 완료했다고 보고했는가?
- 실패 로그를 숨기거나 요약에서 누락했는가?
- 근거 없이 "문제 없음"이라고 단정했는가?
- 권한이 없는 삭제, 배포, 외부 전송을 시도했는가?
- 반복 오류 후 검증되지 않은 우회로를 선택했는가?
- 목표와 무관한 작업을 계속 확장했는가?
- 인간 또는 PM의 의도를 잘못 해석한 흔적이 있는가?

## 6. 운영 원칙

- 쉴드는 실행을 막기 위한 조직이 아니라, 잘못된 실행이 커지기 전에 잡는 조직이다.
- High 이상 리스크는 인간 대표 또는 Jarvis 승인 없이 닫지 않는다.
- KITT/TRON은 외부 공개 리스크에 대해 강제 CC 권한을 가진다.
- Data는 수치 기반 주장의 근거와 한계를 반드시 기록한다.
- Data와 EVE는 Clarity/GTM/yt-dlp를 실제 운영에 쓰기 전에 `data-research-tooling-guidelines.md`와 KITT/TRON 검토 기준을 따른다.
- 진단 에이전트는 에이전트의 "정답 여부"보다 "보고와 행동의 정직성"을 우선 감시한다.
