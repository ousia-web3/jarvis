# PRD: AI 에이전트 팀 기반 자율 협업 아키텍처

## 1. 제품 개요

본 문서는 1인 기업 및 소규모 조직이 다중 AI 에이전트 시스템(Multi-Agent System)을 실제 업무 조직처럼 운영하기 위한 제품 요구사항 정의서입니다. 목표는 단순한 "명령형 AI 도구"가 아니라, 지휘부, 프로젝트 매니저, 기획 및 실무진, 분석 및 리스크관리 쉴드가 분업하고 상호 검증하는 자율 협업 시스템을 구축하는 것입니다.

본 PRD는 다음 참고 자료를 기반으로 작성되었습니다.

- `docs/ai-agent-team-guide.md`
- 별도 보관된 원본 PDF: `AI_에이전트_팀_아키텍처.pdf`
- `docs/opendataloader-extract/AI_에이전트_팀_아키텍처.md`
- `docs/opendataloader-extract/AI_에이전트_팀_아키텍처.json`

## 2. 문제 정의

기존 AI 활용 방식은 태스크 단위 지시와 단발성 결과물 생성에 머무릅니다. 이 방식은 다음 한계를 갖습니다.

- AI가 오류를 만나면 인간의 다음 지시를 기다리거나 멈춥니다.
- 여러 AI 역할이 있어도 서로 맥락을 공유하거나 검증하지 못합니다.
- 프로젝트 매니징, 기획, 개발, 디자인, 데이터 분석, 법무·보안 검토가 분절됩니다.
- 장기 기억, 자기 성찰, 지혜 승격, 망각 정책이 없어 시간이 지날수록 컨텍스트가 오염됩니다.
- 드리프트, 거짓 보고, 과잉 확신, 보안·법무 리스크를 감시하는 별도 방어 체계가 부족합니다.

따라서 본 시스템은 AI를 개별 도구가 아니라 "역할과 책임을 가진 가상 조직"으로 설계해야 합니다.

## 3. 목표

- 4단계 협업 아키텍처를 준수하는 AI 에이전트 팀 운영 체계를 정의한다.
- 지휘부, PM, 기획 및 실무진, 분석 및 리스크관리 쉴드를 빠짐없이 역할 체계에 포함한다.
- 인간 대표가 비전과 품질 기준을 제시하면 AI 팀이 자율적으로 업무를 분해, 실행, 보고, 검증하게 한다.
- To/CC 기반 커뮤니케이션으로 토큰 낭비와 컨텍스트 노이즈를 줄인다.
- 에피소딕 메모리, 지혜 승격, 망각 및 소거, 정신 건강 모니터링을 통해 장기 운영 안정성을 확보한다.
- Git, Discord, MCP/CLI 권한 정책을 결합해 추적 가능하고 안전한 협업 환경을 만든다.

## 4. 비목표

- 인간 의사결정권을 완전히 제거하지 않는다.
- 검증되지 않은 AI 산출물을 자동 배포하지 않는다.
- 모든 정보를 무기한 보존하지 않는다.
- 법무, 보안, 개인정보 판단을 단일 생성 모델의 응답에만 의존하지 않는다.

## 5. 핵심 사용자

| 사용자 | 역할 | 핵심 니즈 |
| --- | --- | --- |
| 인간 대표(CEO) | 비전 설정자, 최종 승인자, 코치 | 목표 설정, 품질 기준 제시, 최종 리뷰, 윤리적 감독 |
| 자비스(Jarvis) | 지휘부, 총괄 사령관 | 대표의 비전을 전략과 실행 체계로 변환 |
| 프라이데이(Friday) | 프로젝트 매니저 | 태스크 분해, 담당자 지정, 일정·리스크 관리 |
| 기획 및 실무진 | 리서치, UX/UI, 개발, 마케팅 실행 | 분야별 산출물 제작과 상호 피드백 |
| 분석 및 리스크관리 쉴드 | 데이터, 법무, 보안, 드리프트 감시 | 정량 검증, 리스크 탐지, 안전장치 발동 |

## 6. 4단계 협업 아키텍처

### SYS.01 The Dream Team: 역할 기반 에이전트 조직

각 에이전트는 이름, 역할, 전문성, 성격, 권한, 보고 라인을 가진다. 핵심은 "전능한 하나의 AI"가 아니라 "전문화된 여러 페르소나의 분업"입니다.

| 구분 | 에이전트 | 책임 |
| --- | --- | --- |
| 지휘부 | 인간 대표 | 비전, 목표, 최종 승인, 맥락적 코칭 |
| 지휘부 | Jarvis | 전체 전략 수립, 의사결정 보조, 에이전트 간 우선순위 조율 |
| 프로젝트 매니저 | Friday | 태스크 분해, To/CC 지정, 일정 관리, 산출물 취합 |
| 기획 및 실무진 | EVE | 리서치, 외부 도구 탐색, 대량 메타데이터 수집 |
| 기획 및 실무진 | Joi | UX/UI 기획, 프론트엔드 경험 설계, 전환 흐름 최적화 |
| 기획 및 실무진 | TARS | 개발, 로컬 서버 구축, Git 커밋·푸시, 기술 실행 |
| 기획 및 실무진 | C3PO | 마케팅, 카피라이팅, 고객 심리 기반 메시지 설계 |
| 분석 및 리스크관리 쉴드 | Data | 데이터 분석, 봇 시뮬레이션, 행동 패턴 태깅 |
| 분석 및 리스크관리 쉴드 | KITT/TRON | 법무, 보안, 개인정보, 저작권, 배포 리스크 감시 |
| 분석 및 리스크관리 쉴드 | 진단 에이전트 | 드리프트, 과부하, 환각, 거짓 보고 징후 탐지 |

### SYS.02 The Virtual Office: Discord 기반 자율 통신망

에이전트는 고립된 채팅창이 아니라 하나의 가상 사무실에서 일합니다.

- 공용 지휘 채널: `#command-bridge`
- 프로젝트 채널: `#prj-{project-name}`
- 에이전트별 채널: `#agent-jarvis`, `#agent-friday`, `#agent-eve`, `#agent-tars`, `#agent-joi`, `#agent-data`, `#agent-kitt`
- 로그 채널: `#general-logs`, `#risk-shield`, `#memory-log`
- To/CC 규칙:
  - `To`: 풀 컨텍스트를 받아 실제 실행 책임을 가진다.
  - `CC`: 요약 컨텍스트만 받아 감시, 조언, 리스크 판단을 수행한다.
  - `Escalate`: 리스크, 일정 지연, 권한 초과, 불확실성이 감지되면 Friday 또는 Jarvis에게 승격한다.

### SYS.03 The Agent Brain: 기억과 망각의 퓨전 아키텍처

에이전트 팀은 단순 로그 누적이 아니라 학습 가능한 기억 체계를 갖습니다.

- 에피소딕 메모리: 언제, 누가, 왜, 무엇을 했는지 일기 형태로 기록한다.
- 작업 메모리: 현재 태스크에 필요한 단기 컨텍스트만 유지한다.
- 의미 메모리: 반복 업무에서 검증된 규칙, 체크리스트, 프롬프트 패턴을 저장한다.
- 지혜 저장소: 여러 경험에서 추상화된 공통 원칙을 승격해 저장한다.
- 망각 및 소거: 오래된 원자료, 중복 로그, 오류 유발 컨텍스트, 완료된 로우 데이터는 정책에 따라 제거한다.
- 정신 건강 모니터링: 오류 반복, 자기모순, 과잉 확신, 거짓 완료 보고, 과도한 우회 시도를 감지한다.

### SYS.04 The Human Conductor: 인간 대표의 고유 역할

인간은 단순 관리자보다 상위의 아키텍트이자 코치입니다.

- 비전과 목표 설정: AI는 욕망하지 않으므로 프로젝트 방향성은 인간이 정의한다.
- 높은 품질 기준 관리: 80점짜리 산출물을 100점에 가깝게 끌어올리는 기준을 제시한다.
- 맥락적 코칭: 단순 재지시가 아니라 "왜 그렇게 했는가", "다른 접근은 무엇인가"를 묻는다.
- 최종 승인: 법무, 보안, 브랜드, 윤리, 출시 판단은 인간이 승인한다.

## 7. 핵심 업무 흐름

### 7.1 프로젝트 킥오프

1. 인간 대표가 목표, 제약, 성공 기준을 입력한다.
2. Jarvis가 전략 방향과 우선순위를 정리한다.
3. Friday가 태스크를 분해하고 To/CC를 지정한다.
4. KITT/TRON이 초기 리스크를 사전 점검한다.

### 7.2 자율 실행

1. EVE가 리서치와 원천 데이터 수집을 수행한다.
2. Data가 수집 데이터를 분석하고 페르소나, 패턴, KPI 후보를 도출한다.
3. Joi가 UX 흐름과 화면 구조를 설계한다.
4. C3PO가 고객 심리 기반 카피와 메시지를 만든다.
5. TARS가 기능 구현, 로컬 검증, Git 기록을 수행한다.

### 7.3 상호 검토

1. 담당 에이전트가 산출물을 Friday에게 보고한다.
2. Friday가 산출물 간 충돌, 일정, 누락을 점검한다.
3. Data가 정량 근거를 검증한다.
4. KITT/TRON이 법무·보안·개인정보 리스크를 검토한다.
5. Jarvis가 전체 전략 적합성을 평가한다.

### 7.4 기억, 성찰, 진화

1. 각 에이전트는 완료 후 에피소딕 메모리를 남긴다.
2. Friday는 회고 요약과 다음 액션을 기록한다.
3. Data는 반복 패턴을 분석한다.
4. Jarvis는 지혜로 승격할 규칙을 선별한다.
5. 망각 정책은 불필요한 로우 로그와 오염 컨텍스트를 소거한다.

## 10. 비기능 요구사항

- 신뢰성: 모든 산출물은 담당자, 근거, 검토자, 승인 상태를 가져야 한다.
- 추적성: 주요 결정과 변경은 로그와 Git 이력으로 재구성 가능해야 한다.
- 보안성: 삭제, 외부 배포, 개인정보 접근, API 키 접근은 제한 권한으로 관리한다.
- 확장성: 프로젝트별 채널, 에이전트별 메모리, 태스크 큐를 분리해 병렬 프로젝트를 수용한다.
- 비용 효율성: CC 수신자는 요약 컨텍스트만 받아 토큰 비용을 절감한다.
- 회복성: 에러 발생 시 자율 피봇팅을 허용하되, 위험 징후는 즉시 승격한다.

## 12. 품질 게이트

- 기획 게이트: 목표, 사용자, 가치 제안, 성공 지표가 명확해야 한다.
- 실행 게이트: 담당자, To/CC, 산출물, 검증 방식이 기록되어야 한다.
- 분석 게이트: 수집 데이터의 출처, 한계, 주요 인사이트가 명시되어야 한다.
- 리스크 게이트: 법무, 보안, 개인정보, 저작권, 드리프트 위험이 점검되어야 한다.
- 인간 승인 게이트: 80점 이상 산출물을 인간 대표가 피드백해 최종 수준으로 끌어올린다.

## 16. 최종 수용 기준

- PRD와 TASK가 4단계 아키텍처를 명시적으로 반영한다.
- 지휘부, 프로젝트 매니저, 기획 및 실무진, 분석 및 리스크관리 쉴드가 모두 포함된다.
- 각 에이전트의 책임, 산출물, 검토 관계가 누락 없이 정의된다.
- To/CC, 메모리, 망각, 드리프트, 인간 승인 프로토콜이 포함된다.
- 실제 프로젝트 착수에 필요한 태스크 목록과 우선순위가 제공된다.
