User Manual

AI 에이전트 팀오케스트레이션 사용 매뉴얼

이 매뉴얼은 `docs` 폴더에 구축된 Jarvis AI 에이전트 팀 운영 문서를 실제 프로젝트에 적용하는 방법을 설명합니다. 정식 운영 구조는 에이전트 팀 오케스트레이션(agent-team-orchestration)입니다. 목표는 단순히 문서를 읽는 것이 아니라, 인간 대표가 목표를 입력하고 Jarvis, Friday, 실무진, 리스크 쉴드가 일관된 절차로 움직이도록 만드는 것입니다.

4단계 아키텍처 SYS.01-04: Dream Team, Virtual Office, Agent Brain, Human Conductor
10개 인간 대표 포함 에이전트별 분리 지침
65개+ 65개+ 운영 자산 인벤토리, 저장소, 평가 하네스, 설계 리뷰 스킬
Dynamic Workflow L4 실행 성숙도: Task Graph, Worker Manifest, Parallel Executor, Verifier/Fixer/Aggregator
적용 상태: 이 매뉴얼은 Jarvis AI 에이전트 팀 오케스트레이션을 기준으로 작성되었습니다. 운영 본체는 Jarvis 전략화, Friday 태스크 분해, 역할별 실행, Risk Shield, Human Conductor 승인으로 이어지는 4단계 아키텍처입니다. 대시보드와 이벤트 로그는 흐름을 보여주는 관찰 하네스이고, Dynamic Workflow는 큰 작업을 task별 worker, verifier, fixer, aggregator로 분리하는 실행 하네스입니다. 최신 운영 기준에서는 Done = 검증 + Work Log + Episodic Memory + 운영 게이트 통과로 봅니다.

오케스트레이션 적용 확인

이 프로젝트의 운영 본체는 에이전트 팀 오케스트레이션입니다. 요청을 전략, 태스크, 실행, 검토, 기억, 승인으로 라우팅하는 다중 에이전트 운영 구조이며, 대시보드는 관찰 하네스, Dynamic Workflow는 실행 하네스로 붙습니다.

구조명

Jarvis AI 에이전트 팀 오케스트레이션

핵심 스킬은 agent-team-orchestration입니다.

적용 위치

AGENTS.md, docs/README.md, Simple Start Request가 같은 운영 흐름을 가리킵니다.

운영 본체와 하네스

대시보드와 task-events.jsonl은 관찰 하네스입니다. Dynamic Workflow는 실행 하네스입니다. 의사결정 본체는 역할 라우팅과 승인 흐름입니다.

구분 의미 언제 적용하나
4단계 아키텍처 SYS.01 Dream Team, SYS.02 Virtual Office, SYS.03 Agent Brain, SYS.04 Human Conductor 모든 Jarvis 작업 요청의 기본 운영 구조
Dynamic Workflow L1-L4 Task Graph, Worker Manifest, Parallel Executor, Verifier/Fixer/Aggregator로 구성된 실행 성숙도 큰 작업을 여러 task와 일회성 worker로 나누어 병렬 실행할 때
관찰 하네스 / 실행 하네스 대시보드는 관찰, Dynamic Workflow는 실행을 맡고 운영 본체를 대체하지 않음 진행 상황을 보거나 대량 작업을 분산 처리할 때
오케스트레이션 agent-team-orchestration Jarvis 전략화 Friday 태스크 분해 Risk Shield Human Conductor 승인

한눈에 보는 오케스트레이션 흐름

인간 대표는 목표만 말하고, 에이전트 팀은 아래 순서로 일을 나눠 실행합니다. 처음 보는 사람도 왼쪽에서 오른쪽으로 읽으면 전체 운영 구조를 파악할 수 있습니다.

Human1
작업 요청

하고 싶은 일을 짧게 말합니다. 복잡한 양식은 에이전트 팀이 초안으로 바꿉니다.

Jarvis2
전략화

목표, 성공 기준, 우선순위, 막히는 리스크를 먼저 정리합니다.

Friday3
태스크 분해

Owner(To), CC, 산출물, 완료 기준을 정해 일이 흩어지지 않게 합니다.

Dream Team4
실무 실행

EVE, Joi, TARS, C3PO가 리서치, UX, 구현, 카피를 맡습니다.

Shield5
검토와 방어

Data, KITT/TRON, Diagnostic Agent가 근거, 보안, 드리프트를 확인합니다.

Brain6
기억과 승인

Work Log와 회고를 남기고, 공개나 큰 결정은 Human Conductor가 승인합니다.

읽는 법: 평상시에는 1번부터 6번까지 자동으로 흐르고, 삭제·배포·개인정보·결제·법무/보안처럼 되돌리기 어려운 결정만 인간 대표에게 멈춰서 확인합니다.

PUML 소스 흐름도

아래 PlantUML 소스는 문서나 위키에 그대로 붙여 시각 흐름도로 렌더링할 수 있습니다. 핵심은 요청이 전략화, 분해, 실행, 리스크 쉴드, 기억, 승인으로 닫힌다는 점입니다. 

@startuml
title Jarvis AI Agent Team Orchestration Flow

start
:Human Conductor\n작업 요청 또는 Human Brief 입력;
:Jarvis\n목표, 성공 기준, 우선순위, 차단 리스크 전략화;
:Friday\nOwner(To), CC, 산출물, 완료 기준 태스크 분해;

partition "SYS.01 Dream Team" {
  :EVE\n리서치와 자료 수집;
  :Joi\nUX와 화면 경험 설계;
  :TARS\n구현과 기술 검증;
  :C3PO\n카피와 커뮤니케이션 정리;
}

partition "SYS.02 Virtual Office" {
  :To/CC 메시지로 진행 상황 공유;
  :dashboards/task-events.jsonl에 역할별 이벤트 기록;
}

partition "Risk Shield" {
  :Data\n정량 근거와 KPI 검토;
  :KITT/TRON\n법무, 보안, 개인정보, 릴리스 리스크 검토;
  :Diagnostic Agent\n드리프트와 의심스러운 완료 보고 진단;
}

if (차단 리스크가 있는가?) then (예)
  :Human Conductor\n최종 승인 또는 방향 전환;
else (아니오)
  :검증 증거 정리;
endif

partition "SYS.03 Agent Brain" {
  :Work Log 작성;
  :Episodic Memory 기록;
  :반복 가능한 원칙은 Wisdom 후보로 승격;
}

:close-jarvis-request.ps1\nDone 이벤트와 완료 산출물 정리;
stop
@enduml

1. 빠른 시작

사용자는 복잡한 양식을 먼저 채우지 않습니다. 짧은 자연어 요청을 입력하면 Jarvis와 Friday가 Human Brief 초안을 자동 생성하고, 오케스트레이션 흐름에 따라 전략화, 태스크 분해, 실무 실행, 리스크 쉴드까지 이어갑니다.

IDE 창에 넣는 최소 요청

docs 기준으로 에이전트 팀 모드로 진행해줘.
요청: [하고 싶은 일 한 줄]

더 짧게 쓰는 방식

[하고 싶은 일] 해줘. docs의 에이전트 팀 운영 규칙대로 자동 진행해.
  1. 짧은 요청 입력
    인간 대표는 하고 싶은 일을 한 줄로 입력합니다. 자세한 예시는 Simple Start Request를 사용합니다.
  2. 요청 폴더와 대시보드 이벤트 생성
    신규 작업이면 work-requests/YYYY-MM-DD-request-slug/를 만들고, scripts/start-jarvis-request.ps1로 첫 이벤트와 로컬 대시보드를 준비합니다.
  3. Human Brief 자동 생성
    Jarvis와 Friday가 사용자 원문을 Human Brief Template 기준으로 프로젝트명, 목표, 성공 기준, 금지 사항, 승인 조건으로 변환합니다.
  4. 전략 브리프 생성
    Jarvis는 Jarvis Command Protocol을 사용해 목표를 전략, 우선순위, 리스크로 변환합니다.
  5. 태스크 분해
    Friday는 Friday Task Breakdown Template으로 Owner(To), CC, 산출물, 완료 기준을 지정합니다.
  6. 실무 실행
    EVE, Joi, TARS, C3PO가 각자의 에이전트 지침에 따라 리서치, UX, 개발, 카피를 수행합니다.
  7. 분석 및 리스크 쉴드
    Data, KITT/TRON, 진단 에이전트가 정량 근거, 법무·보안, 드리프트를 검토합니다.
  8. 회고와 지혜 승격
    완료 후 Work Log와 Episodic Memory를 남기고, 반복 가능한 원칙은 지혜로 승격합니다. 표준 완료는 scripts/close-jarvis-request.ps1로 닫습니다.
최초 30분 운영 팁: 사용자는 요청만 짧게 입력합니다. 에이전트 팀은 Human Brief, Jarvis Command, Friday Task를 먼저 초안으로 만들고, 법무·보안·개인정보·외부 공개·삭제·배포처럼 위험한 항목만 인간 대표에게 되묻습니다.
완료 기준: 작업이 끝났다고 말하기 전에 scripts/validate-jarvis-request.ps1 -RequestId <request-slug> -Strict가 통과해야 합니다. 완료 훅은 Done 이벤트뿐 아니라 Work Log와 Episodic Memory까지 함께 남기는 것을 기본값으로 삼습니다.

1-1. IDE 자동 적용 규칙

이제 사용자는 매번 “문서 먼저 읽고 4단계 아키텍처와 에이전트 역할대로 움직여라”라고 말하지 않아도 됩니다. 프로젝트 루트의 규칙 파일들이 IDE와 에이전트에게 기본 운영 방식을 알려줍니다.

도구/환경 자동 적용 파일 역할
Codex/공통 에이전트 AGENTS.md 프로젝트 전체 기본 규칙, L0/L1/L2 실행 트랙, 역할 라우팅을 정의하는 SSOT입니다.
Cursor jarvis-agent-team.mdc alwaysApply: true로 AGENTS.md와 실행 모드 가이드를 참조합니다.
Antigravity jarvis-agent-team.md Always On 규칙으로 AGENTS.md를 SSOT로 참조하고 Antigravity 브라우저/프리뷰 규칙을 적용합니다.
GitHub Copilot copilot-instructions.md Copilot Chat 또는 코드 보조가 프로젝트 운영 모델을 참고하도록 합니다.
Windsurf .windsurfrules Windsurf 계열 에이전트가 Human Brief 자동 생성 흐름으로 시작하게 합니다.
Cline/Roo .clinerules, .clinerule 복수형과 단수형 규칙 파일을 모두 제공해 도구별 참조 차이를 흡수합니다.
Jarvis Skill SKILL.md 한글 워크플로우, 쉬운 시작 규칙, 페르소나 라우팅, 가드레일을 담은 핵심 운영 스킬입니다.
Jarvis Design Review SKILL.md 신규 MVP, 고위험 기능, 아키텍처 결정, 7개 문서 산출이 필요할 때만 쓰는 전문가용 설계 리뷰 보조 모드입니다.
운영 기준: 사용자가 한 줄로 요청하면 에이전트는 먼저 Human Brief 초안을 만들고, Jarvis 전략화와 Friday 태스크 분해로 넘어갑니다. 일반적인 문서 작업, 내부 기획, 파일 수정, 비파괴 검증은 승인 없이 진행하고, 삭제·외부 배포·개인정보·비밀키·결제·법무/보안 판단만 확인합니다. 새 운영 훅은 시작(start-jarvis-request.ps1), 검증(validate-jarvis-request.ps1), 완료(close-jarvis-request.ps1)를 분리합니다.

2. 문서 지도

이 프로젝트의 문서는 역할별로 분리되어 있습니다. 아래 표에서 “무엇을 하려는지”를 기준으로 파일을 찾으면 됩니다.

목적 먼저 볼 파일 언제 사용하는가
한 줄로 시작 Simple Start Request IDE 창에 넣을 최소 요청문이 필요할 때
실행 모드 선택 Execution Mode Guide L0/L1/L2, Design Review, IA Draft, SI 리서치 깊이를 고를 때
IDE 자동 적용 확인 AGENTS.md, Cursor Rule, Antigravity Rule, SKILL.md 새 세션에서도 Jarvis 운영 규칙이 기본값으로 적용되는지 확인할 때
전체 구조 이해 README.md 문서 위치와 운영 순서를 빠르게 파악할 때
제품 요구사항 확인 PRD 목표, 기능 요구사항, 품질 게이트, 성공 지표를 확인할 때
구현 및 운영 태스크 확인 TASK 어떤 문서와 운영 체계가 만들어졌는지 추적할 때
제품형 운영 기준 확인 Operating Principles, Architecture Contracts 철학을 실제 판단 규칙과 4단계 입출력 계약으로 연결할 때
요청 완료 가능 여부 확인 Request State Machine Intake부터 Done까지 어떤 게이트를 통과해야 하는지 확인할 때
에이전트 배정 Agent Management Index 누구를 To로 둘지, 누구를 CC로 둘지 결정할 때
기억과 평가 저장소 확인 Memory Store, Evaluation Harness 작업 경험, 지혜 후보, 망각 큐, 반복 평가 기준을 확인할 때
리스크 검토 Risk Shield, Release Risk Gate 외부 공개, 배포, 개인정보, 보안 이슈가 있을 때
PDF 추출 근거 확인 PDF Extraction Report PDF가 어떻게 추출되었고 어떤 제한이 있었는지 확인할 때
파일 관리 기준 확인 File Management Policy 원본, 생성물, evidence, 임시 로그, 아카이브 기준을 확인할 때
작업 요청 건강도 확인 audit-work-requests.ps1 작업 요청 폴더의 README, evidence, outputs 누락을 점검할 때

2-2. 제품형 운영 시스템

최신 Jarvis 운영 기준은 “좋은 철학과 문서”에서 멈추지 않고, 요청이 실제로 닫힐 수 있는지 확인하는 상태 머신과 게이트를 사용합니다. 사용자는 여전히 한 줄로 요청하지만, 내부적으로는 아래 운영 장치가 요청을 추적합니다.

운영 원칙

Operating Principles는 Owner(To), CC, 리스크, 검증, 기억, 로컬 우선 원칙을 판단 규칙으로 고정합니다.

4단계 계약

Architecture Contracts는 Dream Team, Virtual Office, Agent Brain, Human Conductor의 입력과 출력을 연결합니다.

요청 상태 머신

Request State Machine은 Intake, Strategy, Dispatch, Execution, Review, Verification, Memory, Done을 완료 순서로 둡니다.

필수 완료 게이트

게이트 무엇을 확인하는가 주요 증거
Intake 요청 ID와 첫 이벤트가 있는가 dashboards/task-events.jsonl, 작업 요청 폴더
Strategy / Dispatch Jarvis 판단과 Friday Owner(To)/CC 배정이 있는가 Human Brief, Friday 분해 이벤트
Execution / Risk 실무 실행과 필요한 리스크 검토가 끝났는가 역할별 이벤트, KITT/TRON 또는 Data 검토
Verification 테스트, 브라우저 확인, 리뷰 결과 같은 증거가 있는가 검증 로그, 스크린샷, 산출물 링크
Memory / Done Work Log, Episodic Memory, 최신 Done 이벤트가 있는가 memory/work-logs/, memory/episodic/, Done 이벤트

운영 훅

시작과 검증

  • scripts/start-jarvis-request.ps1: 대시보드 서버와 첫 이벤트를 준비합니다.
  • scripts/validate-jarvis-request.ps1: 요청 상태 머신 게이트 통과 여부를 점검합니다.
  • scripts/audit-work-requests.ps1: 작업 요청 폴더의 README, evidence, outputs 상태를 점검합니다.

완료와 기억

  • scripts/close-jarvis-request.ps1: Done 이벤트, Work Log, Episodic Memory를 함께 생성합니다.
  • scripts/promote-wisdom.ps1, scripts/purge-memory.ps1: 지혜 승격 후보와 망각 큐를 운영합니다.
대시보드 운영 게이트: Agent Assignment Dashboard는 최신 요청 이벤트를 읽고 필수 게이트 통과 수, 경고, 차단 상태를 보여줍니다. Done으로 보이더라도 검증과 메모리가 빠지면 운영상 닫힌 요청으로 보지 않습니다.

3. 완벽한 협업을 위한 4단계 아키텍처

이 프로젝트는 네 개의 시스템 레이어가 동시에 작동해야 완성됩니다. 한 레이어라도 빠지면 협업은 가능해도 안정적인 자율 협업은 어렵습니다. 최신 운영 기준에서는 Architecture Contracts가 각 레이어의 입력과 출력을 정의하고, Operating Principles가 충돌 상황의 우선순위를 정합니다.

SYS.01 The Dream Team

역할 기반 에이전트 조직입니다. 인간 대표, Jarvis, Friday, 실무진, 리스크 쉴드를 명확히 나누고 각자의 책임과 권한을 분리합니다.

상세 문서 열기

SYS.02 The Virtual Office

Discord 같은 가상 사무실입니다. 채널, 로그, To/CC, 승격 규칙을 통해 에이전트가 고립되지 않고 협업하게 합니다.

상세 문서 열기

기억, 회고, 지혜 승격, 망각을 담당합니다. 단순 로그 누적이 아니라 재사용 가능한 운영 원칙을 만들어냅니다.

상세 문서 열기

SYS.04 The Human Conductor

인간 대표의 고유 역할입니다. 비전, 높은 기준, 맥락적 코칭, 최종 승인이라는 인간만의 책임을 보존합니다.

5. 표준 업무 흐름

모든 프로젝트는 킥오프, 실행, 검토, 기억, 공개 판단 순서로 운영합니다. 급한 프로젝트라도 이 흐름은 생략하지 않는 것이 좋습니다.

Kickoff 요청을 Human Brief, Jarvis 전략, Friday 태스크로 바꿉니다.
Execution 역할별 Owner가 실무를 수행하고 CC는 필요한 부분만 검토합니다.
Review 정량 근거, 보안, 법무, 드리프트를 쉴드 에이전트가 확인합니다.
Memory and Release 로그와 회고를 남기고 외부 공개는 Human Conductor 승인을 거칩니다.

요청 상태 머신

표준 흐름은 Request State Machine으로도 추적합니다. 운영상 Done은 최신 이벤트 상태만이 아니라, 검증과 메모리 게이트까지 통과한 상태입니다. 

Intake
→ Human Brief Draft
→ Jarvis Strategy
→ Friday Dispatch
→ Agent Execution
→ Review
→ Verification
→ Work Log
→ Episodic Memory
→ Done

1단계: Kickoff

  • 사용자 원문에서 Human Brief 자동 초안 생성
  • Jarvis 전략 브리프 작성
  • Friday 태스크 분해
  • KITT/TRON 초기 리스크 확인

2단계: Autonomous Execution

  • EVE 리서치
  • Data 분석
  • Joi UX 설계
  • C3PO 카피 작성
  • TARS 구현

3단계: Cross Review

  • Friday가 산출물 취합
  • Data가 정량 근거 검증
  • KITT/TRON이 법무·보안 검토
  • Diagnostic Agent가 드리프트 점검

4단계: Memory and Release

  • Work Log 작성
  • Episodic Memory 작성
  • 지혜 승격 후보 등록
  • validate-jarvis-request.ps1 검증
  • close-jarvis-request.ps1 완료 처리
  • Release Risk Gate 통과
  • Human Conductor 최종 승인
운영 원칙: 빠른 실행이 필요할수록 Friday의 태스크 분해를 더 짧고 명확하게 작성합니다. 문서를 길게 쓰는 것보다 Owner(To), CC, 완료 기준, 리스크 플래그를 빠뜨리지 않는 것이 중요합니다.

5-1. Dynamic Workflow 실행 레벨 L1-L4

Dynamic Workflow는 기존 Jarvis, Friday, TARS, Data, KITT/TRON 이름을 바꾸는 방식이 아닙니다. 고정 에이전트는 책임과 판단 라인으로 유지하고, 큰 작업을 task 단위로 쪼갠 뒤 일회성 worker를 동적으로 생성해 병렬 실행과 독립 검증을 수행합니다. 여기서 L4는 4단계 아키텍처의 네 번째 단계가 아니라, Dynamic Workflow 실행 하네스의 네 번째 성숙도입니다.

PULM 표기: 이 매뉴얼에서는 PULM을 Previous, Upgrade, Latest, Manual의 약자로 씁니다. 이전 상태, 이번 업그레이드, 최종 적용 상태, 실제 사용법을 한 표에서 확인하기 위한 운영 표기입니다.
PULM 쉽게 말하면 현재 Jarvis 적용
P Previous 이전 버전 고정 역할 에이전트와 task-events.jsonl 기반 이벤트 시각화 중심
U Upgrade 이번 레벨업 L1 Task Graph, L2 Worker Manifest, L3 Parallel Executor, L4 Verifier/Fixer/Aggregator 추가
L Latest 최종 적용 상태 기존 에이전트 이름은 유지하고, 작업 요청 폴더 안에 동적 worker 실행 결과를 보관
M Manual 사용법 new-dynamic-workflow.ps1로 생성하고 run-dynamic-workflow.ps1로 실행/검증/집계

Dynamic Workflow L1-L4 실행 레벨

  1. L1 Task Graph: 작업을 task-001 단위로 쪼갭니다.
  2. L2 Worker Manifest: 각 task에 worker, context packet, result path를 붙입니다.
  3. L3 Parallel Executor: PowerShell job으로 worker를 병렬 실행합니다.
  4. L4 Verifier/Fixer/Aggregator: Data 검증, TARS 수정 검토, Jarvis 집계를 분리합니다.

생성되는 폴더

work-requests/YYYY-MM-DD-request-slug/
  dynamic-workflow/
    task-graph.json
    worker-manifest.json
    packets/
    results/

    aggregation/

  -RequestId "example-request" `
  -MaxParallel 4
안전 기준: 기본 실행은 외부 AI CLI나 임의 명령을 호출하지 않습니다. task 또는 worker에 command를 넣고 실제 명령 실행이 필요할 때만 -AllowCommands를 사용합니다. 삭제, 배포, 외부 전송, 개인정보, 비밀키 처리는 여전히 Human Conductor 확인 대상입니다.

상세 문서는 Dynamic Workflow, 템플릿은 Task Graph와 Worker Manifest를 기준으로 합니다.

6. Jarvis 설계 리뷰 모드

Jarvis 설계 리뷰 모드는 기본 Jarvis 페르소나를 덮어쓰지 않습니다. 짧은 요청을 빠르게 실행하는 기본 흐름은 유지하고, 신규 MVP, 고위험 기능, 아키텍처 결정처럼 결정 품질이 중요한 상황에서만 보조 모드로 사용합니다.

적용 결정: {D-01, Jarvis 페르소나 적용, 기본 덮어쓰기 대신 Design Review Mode로 분리, "기존 자동 실행성과 전문가 설계 검증을 둘 다 살릴 수 있음", "대형/고위험 프로젝트에서 문서 품질과 실행 정확도 상승", "대안: Jarvis 기본 페르소나 전체 교체", "단순 작업에는 비활성화"}

사용할 때

  • 신규 제품, 신규 MVP, 신규 자동화 시스템 기획
  • PRD를 TRD, TASKS, 데이터 설계까지 확장해야 하는 작업
  • 금융, 보안, 개인정보, 결제, 외부 배포가 포함된 고위험 작업
  • 기술 스택, 권한 모델, 데이터 수명주기, 운영 구조를 결정해야 하는 작업

사용하지 않을 때

  • 단순 파일 수정
  • 작은 버그 수정
  • 이미 결정된 스펙의 구현
  • 짧은 문서 정리

운영 규칙

  • 라운드는 R#로 표시하고, 전문가 작업에서는 한 번에 6~10개 질문을 배치로 던질 수 있습니다.
  • 답변이 모호하면 해당 항목은 단일 질문으로 전환해 확정합니다.
  • 2~3라운드마다 결정 요약 5줄, Open Questions 3개, Top Risks 3개를 출력합니다.
  • 결정은 {ID, 항목, 선택, 근거, 영향, 대안, 보류안} 형식의 Decision Log로 남깁니다.
  • PRD, TRD, Flow, ERD, TASKS, 테스트 기준에서 EPIC/FEAT/REQ/NFR/RISK 식별자를 동일하게 유지합니다.

MVP 캡슐

MVP 범위가 확정되면 목표, 타깃 사용자, 핵심 가치 제안, EPIC-1, FEAT-1, 노스스타 지표, 입력 지표, Non-goals, NFR, 데이터 민감도, Top 리스크, 다음 7일 액션을 12줄로 고정합니다. 이 캡슐은 PRD/TRD/TASKS 같은 주요 문서 상단에 반복됩니다.

템플릿은 “작성 양식”이면서 동시에 에이전트 팀의 운영 API입니다. 필드를 채우는 방식이 곧 협업 방식입니다.

8. To/CC 커뮤니케이션

To/CC는 이 프로젝트의 토큰 비용 절감 장치이자 책임 명확화 장치입니다. 모든 사람에게 모든 맥락을 주지 않습니다.

To

실행 책임자입니다. 풀 컨텍스트를 받고 산출물을 만들어야 합니다.

CC

검토자 또는 감시자입니다. 요약 컨텍스트만 받고 필요한 피드백을 제공합니다.

Escalate

상위 판단 신호입니다. Friday, Jarvis, Human Conductor 중 누가 결정해야 하는지 지정합니다.

9. 기억, 성찰, 지혜 승격, 망각

에이전트 팀은 일을 많이 한다고 자동으로 똑똑해지지 않습니다. 경험을 기록하고, 패턴을 추출하고, 지혜로 승격하고, 불필요한 원자료는 소거해야 합니다. 최신 운영 저장소는 memory/ 아래에 Work Log, Episodic Memory, 지혜 후보, 망각 큐를 분리해 둡니다.

단계 사용 문서 담당 목적
업무 로그 work-log-template.md, memory/work-logs/ 모든 에이전트 무엇을 했고 무엇이 남았는지 추적
에피소딕 메모리 episodic-memory-template.md, memory/episodic/ 모든 에이전트 어려움, 피드백, 배운 점 기록
지혜 승격 wisdom-promotion-process.md, wisdom-candidates.md, wisdom-registry.md Data, Jarvis, Friday 반복 가능한 원칙으로 추상화
망각 및 소거 forgetting-and-purging-policy.md, purge-queue.md, quarantine.md KITT/TRON, Data 오래된 로우 데이터와 민감정보 정리

지혜 승격 절차

scripts/promote-wisdom.ps1는 반복 가능한 교훈을 지혜 후보 또는 레지스트리에 기록합니다. 단일 작업의 감상은 바로 지혜가 아니라 후보로 둡니다.

망각 절차

scripts/purge-memory.ps1 -Mode Report는 먼저 삭제 없이 점검합니다. 실제 소거는 민감정보와 보존 필요성을 확인한 뒤 진행합니다.

주의: 모든 로그를 영구 보존하면 시스템이 더 똑똑해지는 것이 아니라 오히려 판단이 흐려질 수 있습니다. 핵심 지혜는 보존하고, 로우 데이터와 중복 로그는 요약 후 소거합니다.

10. 리스크 쉴드 운영

Risk Shield는 속도를 늦추기 위한 장치가 아니라, 잘못된 자동화가 외부로 나가기 전에 잡는 방어 레이어입니다.

리스크 강제 호출 판정 기준
개인정보, 계정, API 키, 비밀정보 KITT/TRON 민감정보가 있으면 외부 공개 금지 또는 수정
저작권, 플랫폼 정책, 외부 콘텐츠 사용 KITT/TRON, EVE 출처, 사용 범위, 공개 가능성 확인
전환율, 수익, KPI 같은 정량 주장 Data 근거, 한계, 신뢰도 명시
반복 실패, 거짓 완료 보고, 목표 이탈 Diagnostic Agent 작업 중지 또는 재검증 승격
외부 공개, 배포, 고객 전달 KITT/TRON, Human Conductor Release Risk Gate 통과

Release Risk Gate 판정

Pass

배포 가능. 치명 리스크가 없고 필수 검토가 끝난 상태입니다.

Pass with Changes

수정 후 배포 가능. 문구, 보안 설정, 고지 사항 보완이 필요합니다.

Blocked

배포 금지. 개인정보, 보안, 법무, 저작권 문제가 해소되지 않았습니다.

11. 파일럿 프로젝트 운영 예시

유튜브 쇼핑몰 파일럿은 이 시스템을 실제로 굴리는 기준 시나리오입니다. 4시간 안에 리서치, 분석, UX, 카피, 개발, 리스크 검토를 압축해서 실행합니다.

12. 유지보수 방법

에이전트 팀 문서는 살아 있는 운영 체계입니다. 프로젝트를 진행하면서 역할, 템플릿, 체크리스트를 개선해야 합니다.

에이전트 추가 방법

  1. ../agents/{agent-name}.md 파일을 새로 만듭니다.
  2. Identity, Mission, Responsibilities, Authority, Boundaries, To/CC Rules, Output Format을 채웁니다.
  3. Agent Management Index에 새 에이전트를 등록합니다.
  4. 필요하면 SKILL.md의 Persona References에도 링크를 추가합니다.
  5. PRD의 에이전트 분리 구조 또는 TASK 산출물 목록을 갱신합니다.

문서 업데이트 규칙

  • 운영 규칙을 바꾸면 관련 템플릿도 함께 갱신합니다.
  • 상태 머신, 완료 기준, 운영 게이트를 바꾸면 request-state-machine.md, validate 훅, 대시보드 표시를 함께 확인합니다.
  • 리스크 정책을 바꾸면 KITT/TRON 지침과 Release Risk Gate를 같이 확인합니다.
  • 데이터 분석 기준을 바꾸면 Data 지침과 Data Analysis Pipeline을 같이 확인합니다.
  • 파일 배치 기준을 바꾸면 File Management Policy, 루트 .gitignore, 폴더별 README를 함께 확인합니다.
  • 실패 사례에서 배운 점은 Wisdom Promotion Process를 통해 지혜 후보로 등록합니다.
  • 운영 품질을 올렸다고 판단하면 evals/의 샘플 태스크와 루브릭으로 실제 완성도를 다시 채점합니다.

  • 보존할 브라우저 스냅샷과 로그만 작업 요청의 evidence/로 복사합니다.
  • docs/opendataloader-extract/는 PDF 추출 보존 자료이며, 최신 설명은 상위 운영 문서에 작성합니다.
  • 작업 요청 건강도는 scripts/audit-work-requests.ps1 -Markdown으로 점검합니다.
  • 삭제, 이동, 대량 이름 변경은 Human Conductor 승인 후 별도 작업 요청으로 진행합니다.

13. 문제 해결

문제 가능한 원인 해결 방법
에이전트가 무엇을 해야 할지 모른다 Human Brief 또는 Friday 태스크가 모호함 목표, 산출물, 완료 기준을 다시 작성하고 Owner(To)를 명확히 지정합니다.
모든 에이전트가 너무 많은 정보를 받는다 CC에게 풀 컨텍스트를 전달함 To/CC 규칙을 적용해 CC에게는 요약 컨텍스트만 전달합니다.
외부 공개 직전 문제가 발견된다 Release Risk Gate를 늦게 호출함 기획 단계부터 KITT/TRON을 CC에 넣고, High 리스크는 즉시 승격합니다.
같은 실수가 반복된다 에피소딕 메모리와 지혜 승격이 누락됨 Work Log와 Episodic Memory를 작성하고 Data가 반복 패턴을 분석합니다.
완료 보고가 실제 산출물과 다르다 드리프트 또는 검증 누락 Diagnostic Agent를 To로 호출하고 Friday, Jarvis, KITT/TRON을 CC에 둡니다.
대시보드에서 Done인데 운영 게이트가 막힌다 검증 증거, Work Log, Episodic Memory 중 하나가 누락됨 validate-jarvis-request.ps1 -RequestId <request-slug> -Strict로 누락 게이트를 확인한 뒤 close-jarvis-request.ps1로 닫습니다.
검색 결과에 브라우저 스냅샷이나 임시 복사본이 섞인다 .playwright-mcp/, tmp/, evidence 덤프가 원본 문서와 함께 검색됨 .gitignore와 파일 관리 정책을 따르고, 필요하면 rg "검색어" -g "!work-requests/**/evidence/**"처럼 제외 검색을 사용합니다.
과거 작업 요청 폴더가 비어 보인다 README, evidence, outputs 설명이 누락됨 scripts/audit-work-requests.ps1 -Markdown으로 건강도를 확인하고, README 또는 안내 파일을 보강합니다.
같은 판단 기준을 매번 다시 설명해야 한다 지혜 후보가 레지스트리로 승격되지 않음 promote-wisdom.ps1로 후보를 등록하고 Data/Jarvis/Friday가 반복 가능성을 확인합니다.
문서가 너무 많아 어디서 시작할지 모르겠다 문서 구조만 보고 업무 흐름을 보지 않음 작업 요청을 먼저 입력합니다. 에이전트 팀이 Human Brief, Jarvis Command, Friday Task 순서로 자동 초안을 만듭니다.
IDE가 매번 “문서를 먼저 읽으라”는 지시를 요구한다 새 세션에서 프로젝트 규칙 파일을 로드하지 못했거나 워크스페이스 루트가 다름 워크스페이스 루트가 jarvis인지 확인하고, AGENTS.md, .cursor/rules/jarvis-agent-team.mdc, .agents/rules/jarvis-agent-team.md, .clinerules, .clinerule이 있는지 확인한 뒤 세션을 새로 시작합니다.

14. 운영 체크리스트

프로젝트 시작 전

  • 요청 슬러그와 작업 폴더가 정해졌는가?
  • start-jarvis-request.ps1로 대시보드 첫 이벤트를 남겼는가?
  • 사용자 원문에서 Human Brief 초안이 생성되었는가?
  • Jarvis 전략 브리프가 있는가?
  • Friday 태스크 분해에 Owner(To)와 CC가 있는가?
  • KITT/TRON 초기 리스크 확인이 필요한가?
  • Data가 검증해야 할 수치 주장이 있는가?

실행 중

  • 각 에이전트가 자기 지침 파일을 기준으로 일하는가?
  • CC에게 필요한 요약만 전달했는가?
  • 우회 또는 피봇이 기록되었는가?
  • 반복 실패가 3회 이상 발생하지 않았는가?
  • 리스크 플래그가 Friday에게 보고되었는가?

완료 후

  • validate-jarvis-request.ps1 -Strict가 통과했는가?
  • 대시보드 운영 게이트가 차단 없이 통과했는가?
  • Work Log가 작성되었는가?
  • Episodic Memory가 작성되었는가?
  • 지혜 승격 후보가 있는가?
  • 로우 데이터 소거 또는 보존 정책을 적용했는가?
  • close-jarvis-request.ps1로 Done 이벤트를 남겼는가?

외부 공개 전

  • Release Risk Gate가 작성되었는가?
  • KITT/TRON 판정이 Pass 또는 Pass with Changes인가?
  • 개인정보와 비밀정보가 제거되었는가?
  • 검증되지 않은 수익·성과 보장이 없는가?
  • Human Conductor 최종 승인이 있는가?
